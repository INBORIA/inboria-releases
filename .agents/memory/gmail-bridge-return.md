# Pont Gmail ↔ Inboria

## Aller (Gmail → Inboria)
L'add-on ouvre l'app avec `?from=gmail` (+ `emailId` si le mail est résolu). Le bouton « ↗ Ouvrir Inboria » existe sur DEUX cartes de l'add-on : la carte d'un mail ouvert ET la carte d'accueil (`onHomepage`/`buildAskCard_`) — important : sans cet ajout, on ne peut ouvrir Inboria qu'en ayant un mail ouvert. La carte d'accueil réutilise `handleOpen_`, qui gère `emailId=null` → ouvre `/dashboard?from=gmail`.

## Retour (Inboria → Gmail), 2 niveaux
1. **Détail d'un mail** : bouton « Retour à Gmail » qui ouvre le **mail EXACT** via `#search/rfc822msgid:<ID>`. Gmail a un deep-link fiable vers un message précis — **Outlook n'en a pas** (différence clé entre les deux ponts).
2. **Réception / partout** : bandeau `GmailReturnBanner` (miroir de `OutlookReturnBanner`, monté dans `dashboard-layout`) — utile car on atterrit souvent en Réception quand le mail n'est pas résolu dans Inboria.

## Pièges durables
- **Le bandeau retire `?from` de l'URL au montage** (replaceState) et persiste un flag `sessionStorage["inboria.fromGmail"]`. Donc tout code qui dépendait de `?from=gmail` (ex. le bouton du détail) DOIT lire le flag sessionStorage en plus du param URL, sinon il disparaît après le strip. Le bandeau ne supprime QUE `from` (jamais `emailId`).
- **Retour onglet** : l'app est ouverte dans un nouvel onglet par l'add-on → `window.close()` pour rendre le focus à Gmail. `window.close()` peut être refusé (onglet non « opened by script ») ; fallback = naviguer Gmail **dans le même onglet** (`location.href`), seulement si `!window.closed` après court délai — jamais `window.open` (popup blocker).
- **Résolution mail Gmail→Inboria : préférer l'ID NATIF, pas le Message-ID RFC822.** `handleOpen_` lisait le Message-ID via `getHeader_` = un appel UrlFetchApp SÉPARÉ à `gmail.googleapis.com` qui échoue silencieusement (renvoie "") → `resolveEmailId_` court-circuite SANS jamais toucher le backend (symptôme : ZÉRO requête `/api/inboria/resolve-email` côté serveur alors que l'utilisateur clique). Fix durable : l'add-on a TOUJOURS `e.gmail.messageId` (ID natif, 0 appel API) → l'envoyer en `nativeMessageId`. Le backend matche `emails.external_id` (`gmail:<id>` / `outlook:<id>`, sinon suffixe `%:<id>`). RFC822 (`provider_message_id`) reste un bonus best-effort. **Why :** un add-on qui dépend d'un appel API tiers fragile pour construire un deep-link tombera en fallback Réception dès que cet appel échoue (scope/token). **How to apply :** pour tout deep-link add-on→app, router par l'identifiant déjà présent dans l'event, jamais par un header nécessitant un 2e appel.
- **`e.gmail.messageId` n'est PAS l'ID hex de l'API Gmail — c'est le format interne `msg-f:<décimal>`** (variantes `msg-a:` / `msg-d:`). Or `auto-sync` stocke `external_id = "gmail:<hex>"` (ex. `gmail:19e89521d41e9c0a`). Le nombre décimal après `msg-f:` EST cet ID hex converti en base 10 (`BigInt("1866906018070305802").toString(16) === "19e89521d41e9c0a"`). Symptôme : la requête `/api/inboria/resolve-email` arrive bien (`hasNative:true`) mais `resolvedId:null` / `matchedBy:null` → atterrissage Réception. **Fix durable (100% backend, AUCUN redéploiement add-on) :** dans la route, générer les variantes de `nativeId` — brut, décimal nu, et `BigInt(décimal).toString(16)` — puis matcher `gmail:<v>`/`outlook:<v>` (exact `.in`) sinon suffixe `%:<v>` par variante. **Why :** convertir côté serveur évite un nouveau déploiement Apps Script chez l'utilisateur (douloureux) ; la conversion décimal→hex est exacte et vérifiée contre la DB. **How to apply :** ne jamais supposer que `e.gmail.messageId` == ID API Gmail ; toujours dériver l'hex avant lookup.
- **Source du Message-ID** = `emails.provider_message_id` (header RFC822, stocké avec OU sans chevrons → strip `<>` + encode). Règle durable : `GET /api/emails/:id` construit un objet de réponse **camelCase explicite** (pas de passthrough `select *`) — tout champ DB utile au front doit être ajouté à ce mapping.
- **Le garde d'auth (App.tsx `ProtectedRoute`) effaçait `?emailId` sur session froide.** Cause racine FINALE du « atterrit en Réception » : quand l'add-on ouvre un onglet NEUF sans session Supabase active sur `/dashboard?emailId=123`, `!session → <Redirect to="/login">` (wouter replaceState) **drop la query string** ; puis `/login` une fois authentifié `→ <Redirect to="/dashboard">` nu → Réception, mail perdu. Le backend résolvait pourtant parfaitement (`matchedBy:native-exact`) — symptôme trompeur : ZÉRO requête `/api/emails/<id>` côté serveur alors que le resolve réussit. **Fix durable :** mémoriser la destination complète (`pathname+search`) dans `sessionStorage["inboria.returnTo"]` AVANT le redirect login (`captureReturnTo`), et après auth rediriger vers elle via `<AuthedHome>` (composant qui consomme returnTo) branché sur les routes `/` ET `/login` à la place de `<Redirect to="/dashboard">`. Le cas session-chaude/persistée n'est pas affecté car `useAuth.loading` reste `true` jusqu'à résolution de `getSession` (le garde renvoie `null`, pas de redirect, tant que loading). **Why :** un deep-link externe (add-on, email, lien partagé) frappe souvent un contexte navigateur sans session → tout garde qui redirige vers login DOIT préserver l'URL de retour, sinon l'intention (quel mail ouvrir) est perdue. **How to apply :** pour toute SPA avec auth-guard + deep-links, capturer returnTo avant le redirect login et le restaurer après, jamais rediriger en dur vers la home.
- **Le returnTo dans l'auth-guard NE SUFFIT PAS — capter le deep-link au boot (main.tsx) est la seule approche fiable.** Suite au fix returnTo (ci-dessus), le mail ne s'ouvrait TOUJOURS pas : sur un onglet neuf, la réhydratation de session Supabase est lente (warnings gotrue « Lock not released within 5000ms / orphaned lock ») et provoque PLUSIEURS redirects/remontages du Dashboard (4 bootstraps observés sur ~23s) ; entre `<AuthedHome>` qui consomme returnTo et `redirectAfterAuth` de login.tsx qui le consomme AUSSI, le dernier nav gagne → `/dashboard` nu, `?emailId` perdu. Symptôme : resolve réussit (`resolvedId:15383`), mais ZÉRO requête `/api/emails/15383` (le Dashboard ne monte jamais avec l'emailId). **Fix durable :** dans `main.tsx`, AVANT tout React/auth, lire `window.location.search` (qui a encore `?emailId`) et stocker `sessionStorage["inboria.pendingEmailId"]={id,ts}` (TTL 2 min). Le Dashboard LIT ce pending (peek, SANS supprimer) si l'URL a perdu l'emailId — à l'init de `selectedEmailId` ET dans l'effet sur changement d'URL. **Why :** le point d'entrée du module s'exécute une fois au chargement, AVANT que n'importe quel garde/redirect puisse toucher l'URL ; c'est le seul instant garanti où la query est intacte. La danse multi-redirects de l'auth est trop imprévisible pour fiabiliser via returnTo seul. **How to apply :** pour tout deep-link externe qui transporte un paramètre critique vers une SPA avec auth-guard, capturer ce paramètre dès l'entrée du bundle (avant React) et le restaurer côté page, plutôt que de compter sur la préservation de l'URL à travers les redirects de login. (Le returnTo reste en place comme bonus pour préserver l'URL complète, mais l'emailId ne dépend plus de lui.)
- **CONSUME-ONCE était un piège : le pending DOIT être lu en peek (sans suppression) et nettoyé SEULEMENT une fois le détail du mail réellement chargé.** Cause racine du « ça ne s'ouvre toujours pas » même après capture au boot : la réhydratation Supabase sur onglet neuf MONTE le Dashboard PLUSIEURS fois (2 bootstraps observés, locks gotrue ~5s). Un `removeItem` à la 1ʳᵉ lecture vide le pending → le 2ᵉ montage ne trouve plus rien → mail perdu (resolve réussit `native-exact`, mais ZÉRO `/api/emails/<id>`). **Fix validé Gmail+Outlook :** `peekPendingEmailId()` (lit + applique TTL, ne supprime jamais) à l'init et dans l'effet URL ; `clearPendingEmailId()` appelé depuis `useEffect(()=>{ if (emailDetailData) clearPendingEmailId(); }, [emailDetailData])` — donc on ne purge qu'au succès réel du fetch détail, ce qui survit aux remontages. **Why :** tout deep-link qui atterrit dans une SPA avec auth-guard subit des remontages imprévisibles ; un état « one-shot » est consommé par un montage perdant avant le montage gagnant. **How to apply :** stocker l'intention de manière idempotente (peek + TTL), ne la consommer qu'à la confirmation de l'effet désiré (détail chargé), jamais à la 1ʳᵉ lecture.
- **IMAP : le backend `resolve-email` couvre DÉJÀ l'IMAP** — la synchro IMAP (`auto-sync.ts` ~L1651) stocke `provider_message_id = envelope.messageId || headers["message-id"]` (RFC822), et l'étape 1 de resolve-email matche dessus (variantes chevrons). `external_id` IMAP = `imap:<email>:<uid>` (la branche native ne teste que `gmail:`/`outlook:`, mais l'étape RFC822 suffit). Donc tout client capable de fournir le Message-ID RFC822 (ex. add-in Outlook avec un compte IMAP branché → `item.internetMessageId`) ouvre déjà le bon mail. **Le vrai manque IMAP = point d'entrée client** : l'IMAP n'a pas de magasin d'add-ons (webmails tiers Orange/OVH/Roundcube, Thunderbird, Apple Mail). Décision produit requise avant de coder.
- **Limite assumée** : lien Gmail sur `mail/u/0` (1er compte). Multi-compte non garanti ; option produit choisie = recherche par Message-ID.
